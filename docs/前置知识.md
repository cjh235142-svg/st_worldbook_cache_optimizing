## 1. DeepSeek 缓存机制

### 1.1 参考资料

- 官方文档：https://api-docs.deepseek.com/guides/kv_cache

### 1.2 核心原理

DeepSeek 的「上下文缓存硬盘技术」对所有用户默认启用。每次请求都会触发硬盘缓存构建。如果后续请求与先前请求有**前缀重叠**，重叠部分将从缓存中获取，计为一次「缓存命中」。

### 1.3 缓存持久化与命中规则

缓存命中要求对应的前缀已经被**持久化**（写入硬盘缓存）。每个缓存前缀是一个**独立、完整**的单元。后续请求必须**完全匹配**一个「缓存前缀单元」才能命中。

**缓存前缀持久化的三种时机**：

1. **请求边界持久化**：每次请求在 **用户输入结尾位置** 和 **模型输出结尾位置** 各生成一个缓存前缀单元。后续请求若完全匹配该单元，即可命中。
2. **公共前缀检测持久化**：系统检测到多个请求有公共前缀时，将该公共前缀作为独立单元持久化。
3. **固定token间隔持久化**：对长输入/长输出，系统以固定token间隔切分缓存前缀单元，避免长前缀因从未到达结束位置而无法命中。

### 1.4 示例

**示例1：多轮对话**

```
第一轮: [system] + [user: "What is the capital of China?"]
第二轮: [system] + [user: "What is the capital of China?"] + [assistant: "北京"] + [user: "What about US?"]
→ 第二轮完全匹配第一轮的缓存前缀单元 → 缓存命中
```

**示例2：长文本QA**

```
第一轮: [system: 你是财务分析师] + [user: <财报全文>\n请总结]
第二轮: [system: 你是财务分析师] + [user: <财报全文>\n请分析盈利能力]
→ 前两轮不命中。系统检测到公共前缀 (system + <财报全文>)，将其持久化
第三轮: [system: 你是财务分析师] + [user: <财报全文>\n请分析收入比]
→ 第三轮完全匹配持久化的公共前缀 → 缓存命中
```

### 1.5 缓存命中状态查看

API 响应的 `usage` 字段包含：
- `prompt_cache_hit_tokens`：本次请求输入中命中缓存的token数
- `prompt_cache_miss_tokens`：本次请求输入中未命中缓存的token数

### 1.6 缓存关键约束

1. **缓存系统是「尽力而为」的**，不保证 100% 命中率
2. 缓存构建需要数秒
3. 不再使用的缓存会在数小时到数天内自动清除
4. **缓存要求前缀完全匹配**，前缀中任何位置的内容变化都会导致缓存断裂

## 2. 酒馆运行逻辑与提示词构建

### 2.1 参考资料

- 酒馆主站：https://sillytavern.app
- 酒馆文档：https://docs.sillytavern.app/
- 酒馆源码：https://github.com/SillyTavern/SillyTavern
- 提示词构建文档：https://docs.sillytavern.app/usage/prompts/
- Prompt Manager 文档：https://docs.sillytavern.app/usage/prompts/prompt-manager/
- World Info 文档：https://docs.sillytavern.app/usage/core-concepts/worldinfo/
- 宏文档：https://docs.sillytavern.app/usage/core-concepts/macros/
- STscript 文档：https://docs.sillytavern.app/usage/st-script/

### 2.2 酒馆架构概述

SillyTavern 是一个本地安装的 LLM 前端界面，支持多种 API 后端。它本身不做推理，只负责组装提示词并发送给后端 API。

酒馆支持两种主要的提示词构建模式：
- **Chat Completion API**：使用 Prompt Manager
- **Text Completion API**：使用 Advanced Formatting + Context Template

由于 DeepSeek 使用 Chat Completion API，本文档聚焦于 Chat Completion 模式。

### 2.3 Prompt Manager 提示词构建流程

Chat Completion 模式下，提示词由 Prompt Manager 中可拖拽排序的提示词列表决定。**越靠近列表顶部的提示词越早发送**，越靠近底部越晚发送。

默认提示词发送顺序（从前到后）：

```
1. Main Prompt (System Prompt)
   → 系统提示词，通常为 "Write {{char}}'s next reply..."
2. World Info (before)        ← position=0 的条目插入此处
3. Persona Description
4. Character Description
5. Character Personality
6. Scenario
7. World Info (after)         ← position=1 的条目插入此处
8. Chat Examples (示例对话)
   → position=5/6 的条目插入示例对话前后
9. Chat History (聊天历史)
   → position=4 (atDepth) 的条目嵌入聊天历史的指定深度
10. Post-History Instructions
```

### 2.4 酒馆消息处理流程（一次生成请求）

```
1. 用户输入消息
   ↓
2. 酒馆收集所有上下文源 (World Info、角色卡、预设等)
   ↓
3. 扫描聊天历史 → 匹配 World Info 关键词 → 激活对应条目
   ↓
4. 按 order 排序已激活的条目 → 根据 position 插入对应位置
   ↓
5. Prompt Manager 按顺序拼接所有部分
   ↓
6. 处理宏替换 ({{char}}, {{user}}, {{getvar::}} 等)
   ↓
7. 发送给 LLM API
   ↓
8. 接收响应 → 渲染到聊天界面
```

### 2.5 World Info 激活与评估

酒馆在每次生成请求时评估 World Info：

1. **扫描阶段**：从聊天历史末尾向前扫描（由 Scan Depth 控制扫描多少条消息）
2. **匹配阶段**：每条消息都检查是否包含已激活 World Book 中条目的关键词
3. **激活阶段**：匹配的条目被加入激活列表
4. **递归阶段**：如果启用了递归扫描，已激活条目的内容会再次被扫描，可能触发更多条目
5. **排序与插入**：所有激活条目按 `order` 排序后按 `position` 插入提示词

## 3. 酒馆世界书格式详解

### 3.1 文件结构

世界书以 JSON 文件存储，顶层结构为：

```json
{
  "entries": {
    "0": { /* 条目对象 */ },
    "1": { /* 条目对象 */ },
    "...": { }
  }
}
```

- `entries` 是一个**字典**，key 为字符串形式的 UID（整数字符串），value 为条目对象
- UID 从 0 起递增分配（见源码 `world-info.js` 中 `getFreeWorldEntryUid`）
- 可能包含可选的 `originalData` 字段（见下一节）

### 3.2 originalData 字段详解

`originalData` 是角色卡 **Spec v2** 格式的嵌入式世界书数据，用于向后兼容。

#### 来源

当角色卡（PNG/JSON）内嵌了 Spec v2 格式的世界书时，酒馆将其转换为内部字典格式（`entries`）的同时，把原始数据原封不动保留在 `originalData` 中，以确保修改后的世界书能以相同的 Spec v2 格式写回角色卡。

手动创建或独立导入的世界书文件通常**不包含**此字段。`originalData` 仅用于导入导出兼容，不参与提示词构建，对缓存优化无影响。

#### 完整结构

```json
{
  "entries": { /* 内部格式, 字典 */ },
  "originalData": {
    "entries": [
      {
        "id": 0,
        "keys": [],
        "secondary_keys": [],
        "comment": "标题",
        "content": "内容文本...",
        "constant": false,
        "selective": true,
        "insertion_order": 100,
        "enabled": false,
        "position": "after_char",
        "use_regex": true,
        "extensions": {
          "position": 6,
          "display_index": 101,
          "probability": 0,
          "useProbability": true,
          "depth": 4,
          "selectiveLogic": 0,
          "outlet_name": "",
          "group": "",
          "group_override": false,
          "group_weight": 100,
          "prevent_recursion": false,
          "delay_until_recursion": false,
          "scan_depth": null,
          "match_whole_words": null,
          "use_group_scoring": false,
          "case_sensitive": null,
          "automation_id": "",
          "role": 0,
          "vectorized": false,
          "sticky": 0,
          "cooldown": 0,
          "delay": 0,
          "match_persona_description": false,
          "match_character_description": false,
          "match_character_personality": false,
          "match_character_depth_prompt": false,
          "match_scenario": false,
          "match_creator_notes": false,
          "triggers": [],
          "ignore_budget": false
        }
      }
    ]
  }
}
```

#### 与内部格式的核心差异

`originalData.entries` 是一个**数组**（不是字典），字段名与内部格式不同：

| Spec v2 (`originalData.entries`) | 内部格式 (`data.entries`) | 说明 |
|---|---|---|
| `id` (number) | `uid` (number, 作字典 key) | 数组索引 vs 字典 key |
| `keys` | `key` | |
| `secondary_keys` | `keysecondary` | |
| `comment` | `comment` | 相同 |
| `content` | `content` | 相同 |
| `constant` | `constant` | 相同 |
| `selective` | `selective` | 相同 |
| `insertion_order` | `order` | **名称不同** |
| **`enabled`** | **`disable`** | **逻辑反转！** `enabled=false` ↔ `disable=true` |
| `position` (string) | `position` (number) | 字符串描述，如 `"after_char"` |
| `extensions.position` | `position` | **数字覆盖值**，为实际生效的 position |
| `extensions.*` (snake_case) | 顶层字段 (camelCase) | 所有扩展字段嵌套在 `extensions` 下 |

#### Spec v2 格式加载代码

当从角色卡 Spec v2 格式加载世界书时，转换逻辑为（`world-info.js:5510-5549`）：

```javascript
// 将 Spec v2 数组条目转换为内部格式对象
const entry = {
    uid: index,
    key: entry.keys,                                                // keys → key
    keysecondary: entry.secondary_keys || [],                       // secondary_keys → keysecondary
    comment: entry.comment || '',
    content: entry.content,
    constant: entry.constant || false,
    selective: entry.selective || false,
    order: entry.insertion_order,                                   // insertion_order → order
    // 优先使用 extensions.position 数字值, 否则从字符串 position 推断

        ?? (entry.position === 'before_char' ? world_info_position.before : world_info_position.after),
    disable: !entry.enabled,                                        // ★ enabled → !disable (反转)
    addMemo: !!entry.comment,
    displayIndex: entry.extensions?.display_index ?? index,
    probability: entry.extensions?.probability ?? 100,
    // ... 所有 extensions.* 字段均映射到顶层
};
```

### 3.3 条目结构（完整字段表）

以下是世界书条目的所有字段。默认值来自源码 `newWorldInfoEntryDefinition`（`world-info.js` L4002-4045）。

#### 3.3.1 核心字段

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `uid` | `number` | 创建时分配 | 唯一整数ID，从0递增 |
| `key` | `string[]` | `[]` | 主要激活关键词，支持 `/regex/` 正则 |

### 3.4 World Info 来源与合并策略

酒馆支持四种不同绑定的 World Info 来源：

| 来源 | 绑定方式 | 合并优先级 |
|---|---|---|
| **Chat Lore** | 绑定到特定聊天 (`chat_metadata['world_info']`) | **最高** (首先插入) |
| **Persona Lore** | 绑定到用户角色 (`power_user.persona_description_lorebook`) | **第二** |
| **Character Lore** | 绑定到角色卡 (主: `character.data.extensions.world`, 附加: `charLore[].extraBooks`) | **第三或第四** |
| **Global Lore** | 全局选择器 (`selected_world_info` 数组) | **第三或第四** (与角色绑定交替) |

三种合并策略（`world_info_character_strategy`）：
- `0` (evenly)：角色+全局混合后统一按 order 排序
- `1` (character_first)：角色先 (按order排)，全局后 (按order排)
- `2` (global_first)：全局先，角色后

去重逻辑：同一个 World Book 名称出现在多个类别时，只保留优先级最高的实例。

### 3.5 常用组织模式与拆分策略

本节分析社区与目标世界书文件中常见的条目内容组织结构，以及如何基于这些结构进行「固定/可变」拆分，以最大化 DeepSeek 缓存前缀命中率。

#### 3.5.1 社区常见组织模式

参考 [World Info Encyclopedia](https://rentry.co/world-info-encyclopedia) 等社区权威指南，常见模式如下：

| 模式 | 说明 | 示例 |
|---|---|---|
| **PList 格式** | 压缩的键值对描述，用 `[]` 包裹 | `[Mossford: town, mossy buildings, has(tavern, bank)]` |
| **Ali:Chat 格式** | 角色对话示例，以 `{{user}}: ...` `{{char}}: ...` 组织 | 用于表达角色对特定事物的态度 |
| **PList Base World** | 两个常驻 `[` 和 `]` 条目包裹所有 PList 条目 | order=2 的 `[` + order=998 的 `]` |
| **递归键值树** | 条目的 content 中包含其他条目的 key，形成递归激活树 | `monsters` 条目内容含 `slimes` → 递归触发 `slimes` 条目 |
| **分组包裹** | 同类条目共享关键词和插入策略 | 城市组、怪物组、角色关系组 |

#### 3.5.2 目标世界书文件结构统计

对 `学园都市0313.json` 进行分析后，识别到以下内容模式：

| 模式 | 在文件中的出现次数 | 说明 |
|---|---|---|
| **EJS 模板块** `<%...%>` | 1716 | 条件逻辑，使用 `getvar('stat_data.$天数')` 等运行时变量 |
| **Markdown H2** `##` | 2672 | 层级标题组织结构 |
| **XML 风格包裹标签**（如 `<背景事件>`） | 104 | 中文命名的语义包裹标签，成对使用 |
| **Hash 标记** `#` (H1-H3) | 4370 | 各级 Markdown 标题 |
| **Order 分布** | 1–500+ | 不同条目的 order 值跨度大，从 1 到数百不等 |

#### 3.5.3 识别到的具体组织模式

**模式 1：XML 标签包裹 + Markdown 分节 + EJS 条件**

条目 "事件：背景" (uid=1)：
```text
<背景事件>                           ← XML 包裹层（静态语义标签，始终存在）
# 冷战期间                            ← Markdown H1（静态标题）
    - 美苏两国都在超能力方面投入了...    ← 静态内容
# 1962年                              ← H1（静态标题）
    - 冥土追魂向亚雷斯塔·克劳利提议在... ← 静态内容
<%_ if (getvar('stat_data.$天数') <= 30) { _%>   ← EJS 条件块（动态—取决于游戏天数）
<%_ } else if (getvar('stat_data.$天数') <= 47) { _%>
# 2013年                              ← H1（动态—条件激活）
    - 一方通行被送入特别班级...          ← 动态—仅在满足天数条件时显示
<%_ } else { _%>
...
```

**模式 2：独立 XML 标签 + 多级 Markdown 标题 + 混合内容**

条目 "科学"：
```text
<科学>                               ← XML 包裹标签
# 定义                                ← H1
    - 通过反复实验并观察总结规律...       ← 全部静态
# 分类
    ## 常规科学                        ← H2
        - 遵循客观规律研究...
    ## 异常科学                        ← H2
        - 以木原一族为首的极端研究流派...
<学园都市研究机构>                     ← XML 子标签（可独立成条目）
# 能力开发课程...
```

**模式 3：角色条目 — 固定设定 + 可能含变量的关系状态**

条目 "核心01：一方通行", "核心01：御坂美琴" 等：
```text
<御坂美琴>                            ← XML 包裹（角色标签）
## 人物介绍                            ← 静态标题
    - 学园都市排名第三的Level 5...       ← 固定设定（不变）
## 能力详细                            ← 静态标题
    - 超电磁炮...
## 关系状态                            ← 可能含变量
    - <% ... %>                        ← 动态内容（随剧情变化）
</御坂美琴>
```

**模式 4：纯 EJS 条件分支全覆盖（不可拆分）**

某些条目整体被 EJS 的 `if / else if / else` 全部分支覆盖，没有任何「非条件」的静态内容。这类条目**不可拆分**，因为每个分支块互为替代关系，拆分后无法独立存在。

**模式 5：专用变量/持久化条目**

如 "变量列表"、"占位符-顶端"、"占位符-用户输入" 等。这些条目本身即为变量管理用途，全动态内容，无需拆分但需要放在 order 靠后的位置。

#### 3.5.4 拆分策略

核心原则：**按语义边界拆分，拆分后的条目各自保留必要的封闭标签和标题上下文，合并后语义等同于原始条目。**

##### 可拆分模式矩阵

| 发现模式 | 拆分边界 | 固定部分 | 可变部分 | 拆分后各自保留的上下文 |
|---|---|---|---|---|
| XML 标签包裹条目 | 同一 XML 标签内的 `#`/`##` 标题边界 | 不含 `<%` 的标题及内容 | 含 `<%` 的标题及内容 | **各自独立包裹同一 XML 标签对** |
| 多级 Markdown 层级条目 | H1/H2 之间的边界 | 不含变量的标题区块 | 含变量的标题区块 | 各自保留所属的 H1 上下文 |
| EJS 条件前置条目 | `<% if { %>` 与前面静态内容的分界 | if 之前的静态内容 | if 块内的动态内容 | 静态部分保留前置标签和标题 |
| 角色条目 | "人物介绍" vs "关系状态" 等语义小节 | 固定设定描述 | 变量引用区 | 各自包裹角色 XML 标签 |
| 嵌套 XML 子标签 | 外层标签内的子 XML 标签 | 外层描述内容 | 子标签内容（可能含变量） | 子标签可独立成条目 |

##### 拆分示例

**原始条目**（假设 "核心01：御坂美琴"）：

```text
<御坂美琴>
## 人物介绍
    学园都市排名第三的Level 5超能力者，就读于常盘台中学...
    能力为超电磁炮（Railgun），能够操纵电磁力...
## 关系状态
    当前对一方通行的态度: <%- getvar('misaka_touma_relation') %>
    当前与上条当麻的好感度: <%- getvar('misaka_touma_favor') %>
</御坂美琴>
```

**拆分后**：

| 新条目 | content | 判定 | order 策略 |
|---|---|---|---|
| 条目 A | `<御坂美琴>\n## 人物介绍\n学园都市排名第三的Level 5超能力者...能力为超电磁炮...\n</御坂美琴>` | **静态** | 低 order（靠前 → 进入缓存前缀） |
| 条目 B | `<御坂美琴>\n## 关系状态\n当前态度: <%- getvar(...) %>\n好感度: <%- getvar(...) %>\n</御坂美琴>` | **动态** | 高 order（靠后 → 在缓存断裂点之后） |

> 拆分后 AI 看到的最终提示词内容完全相同（各自包裹 `<御坂美琴>` 标签），语义无变化。区别仅在于人物设定进入了每轮都可命中的缓存前缀，而关系状态在后续轮次中独立变化不影响前面的缓存。

##### 不可拆分的情况

| 情况 | 原因 |
|---|---|
| **纯 EJS 全覆盖**（整个 content 都在 `<% if/else %>` 分支中） | 条件块互为替代关系，拆分后语义断裂 |
| **同一 `##` 小节内含混排** | 同一语义单元中同时有静态和 `<%` 的文本，无法在更细粒度上拆分而不破坏语义 |
| **`<% } else { %>` 等分支连接符** | 这些是 EJS 语法结构的一部分，不能独立成条目 |
| **`<%_` 前缀的空白控制块** | 如 `<%_ _%>` 行，仅用于控制模板输出的格式空白，无实际内容 |

##### 特殊处理：`@@preprocessing` / `[GENERATE]` 条目

对于使用了提示词模板插件的 World Info 条目，如果其标题带有 `[GENERATE:BEFORE]`、`[GENERATE:AFTER]` 等注入前缀，或内容使用了 `@@generate_before` 等装饰器，拆分后：
- 包含**注入控制逻辑**的部分保留在原条目
- 其余纯粹的静态描述内容拆分到新的独立条目
- 新条目可继承原条目的 same group/trigger 设置以保证同步激活

## 5. 提示词模板插件 (ST-Prompt-Template)

### 5.1 概述

- **用途**：将 EJS (Embedded JavaScript) 模板引擎引入酒馆，允许在提示词、角色卡、世界书中使用完整的 JavaScript 逻辑

### 5.2 EJS 模板语法

在酒馆的任何提示词字段中可使用以下语法：

```
<% ... %>       代码块（无输出）

<%- ... %>      输出（不转义，直接作为HTML）
<%# ... %>      注释
```

**示例：条件逻辑**

```ejs
<% if (getvar('affinity', { defaults: 0 }) > 50) { %>
你是我的挚友。
<% } else { %>
我对你还保持警惕。
<% } %>
```

**示例：动态输出变量**

```ejs
角色当前好感度: <%- getvar('affinity') %>
```

### 5.3 内建函数（核心 API）

#### 变量操作
```javascript
getvar(key, options)          // 读取变量
```

### 5.4 内容注入 (Content Injection)

在 World Info 条目的**标题**中使用前缀，控制注入位置：

| 前缀 | 效果 |
|---|---|
| `[GENERATE:BEFORE]` | 注入到发送给LLM的提示词**开头** (仅🔵) |
| `[GENERATE:AFTER]` | 注入到发送给LLM的提示词**末尾** (🔵和🟢) |
| `[RENDER:BEFORE]` | 注入到LLM输出**渲染开始时** (仅🔵) |
| `[RENDER:AFTER]` | 注入到LLM输出**渲染结束时** (🔵和🟢) |
| `[GENERATE:{idx}:BEFORE]` | 注入到第 {idx} 条消息之前 (0=第一条) |
| `[InitialVariables]` | 将条目内容作为初始变量树 (JSON Object) |

### 5.5 装饰器 (Decorators)

在 World Info 条目**内容开头**使用 `@@` 前缀行：

| 装饰器 | 效果 |
|---|---|
| `@@activate` | 视为🔵常驻条目 |
| `@@dont_activate` | 完全禁止激活 |
| `@@generate_before` | 等同 `[GENERATE:BEFORE]` |
| `@@generate_after` | 等同 `[GENERATE:AFTER]` |
| `@@render_before` | 等同 `[RENDER:BEFORE]` |
| `@@render_after` | 等同 `[RENDER:AFTER]` |
| `@@preprocessing` | 在酒馆处理WI之前先由本扩展处理 |
| `@@if 条件` | 条件为 false 时排除此条目 |
| `@@private` | 将条目内容包裹在 `<% { %>...<% } %>` 中 |
| `@@iframe [标题]` | 渲染时将内容包裹在 `<iframe>` 中 |
| `@@message_formatting` | 输出为 HTML 代码 |
| `@@dont_preload` | 打开角色卡时不预加载 |
| `@@always_enabled` | 强制启用特殊条目 |
| `@@only_preload` | 仅在预加载阶段启用 |

示例：
```
@@generate_before
@@if variables.current_stage === 1
当前处于第一阶段。
```

### 5.6 Prompt Injection (`@INJECT`)

在 World Info 条目**标题**中使用 `@INJECT` 精确控制提示词消息的插入位置（需将条目设置为**禁用**状态才生效）：

```
@INJECT pos=位置,role=角色           // 绝对位置
@INJECT target=角色,index=序号,at=前后,role=角色  // 相对目标消息
@INJECT regex=正则表达式,at=前后,role=角色         // 正则匹配
```

**重要**：对于 Gemini、Claude 等 API，系统消息必须放在开头才能生效。ChatGPT/DeepSeek 的 system 消息不严格要求在最前面，但强烈建议。

在 EJS 模板中可直接访问：

## 7. 缓存优化策略分析

### 7.1 核心原理

DeepSeek 缓存基于**前缀完全匹配**：

```
请求A: [前缀P1] [内容X]
请求B: [前缀P1] [内容Y]
→ 请求B 的前缀P1 命中缓存 ✓

请求A: [前缀P1] [内容X]
请求C: [前缀P2] [内容Z]
→ 请求C 不命中 ✗ (前缀不同)
```

### 7.2 酒馆请求中的「不变前缀」与「变化结尾」

在酒馆的一次完整请求中：

```
┌──────────────────────────────────────────────────────┐
│ [不变部分 - 可缓存前缀]                                │
│  ├─ System Prompt (Main Prompt)                      │
│  ├─ WI Before (position=0, order由低到高)              │
│  ├─ 角色定义 (Description, Personality, Scenario)     │
│  └─ WI After (position=1, order由低到高)               │
│                                                      │
│         ★ 缓存断裂边界! ★                             │
│                                                      │
│ [变化部分 - 每轮都会变化]                              │
│  ├─ 示例对话 (position=5/6, 除非被挤出上下文)          │
│  ├─ 聊天历史 (最新的消息在此追加)                       │
│  │   └─ 嵌入的 WI (position=4 atDepth)               │
│  ├─ Author's Note                                    │
│  └─ Post-History Instructions                       │
└──────────────────────────────────────────────────────┘
```

在「不变部分」中，System Prompt、角色定义基本上是**固定的**。而 World Info 的注入内容取决于：
- 哪些条目的关键词被触发
- 触发条目的 `order` 排序
- 被激活条目的实际 `content`

当聊天历史变化时，新消息可能触发不同的 World Info 关键词组合，导致**原本在靠前位置的条目内容发生变化**，从而破坏整个缓存前缀。

### 7.3 优化目标

在尽量不改变原始世界书语义的条件下：
1. **不常变的条目尽量靠前**（分配较低的 order 值）→ 增加缓存前缀长度
2. **可能随剧情推进变化的条目尽量靠后**（分配较高的 order 值）→ 将变化推到缓存边界之后
3. **保持同一 position 组内彼此无依赖关系的条目的相对顺序不变**

### 7.4 「动态条目」判定依据

一个 World Info 条目是否「可能随剧情推进变化」，主要取决于其 `content` 字段中是否包含：

1. **酒馆原生变量宏**：`{{getvar::...}}`, `{{setvar::...}}`, `{{incvar::...}}`
2. **提示词模板插件 EJS 语法**：`<%...%>`, `<%=...%>`, `<%-...%>`
3. **小白X 变量宏**：`{{xbgetvar_yaml_idx::...}}`, `{{getvar::...}}` 中的变量路径
4. **条件逻辑**：`{{if ...}}`, `<% if (...) { %>` 等
5. **具体变量引用**：任何引用 `variables.*`, `getvar(...)`, `setvar(...)` 的内容
6. **时间变化宏**：`{{time}}`, `{{date}}`, `{{idleDuration}}`（每次展开值均不同）
7. **小白X state 块**：`<state>` （变量管理2.0语法）

### 7.5 排序策略

**前提约束**：
- 优化后必须保证条目间的原始相对顺序不变
- 由于酒馆的最终排序受预设配置、插入深度等多重因素影响，我们无法在运行时完全控制。优化所做的只是基于一个统一的排序标准重新分配 `order` 值，让静态条目尽量靠前、动态条目尽量靠后

**核心思路：三层层级划分**

由于我们以最大化 DeepSeek 缓存命中率为目标，不限制发送给 API 的聊天历史长度（DeepSeek 可缓存任意长度前缀），提示词中的条目可划分为三个层级：

```
layer -1 (缓存前缀核心区) → layer 0 (中间过渡区) → layer 1 (变化区)
   ↑ 最先发送, 高度稳定             ↑                  ↑ 最后发送, 每轮易变
```

- **layer -1**：内容高度稳定，位于缓存前缀安全区内，多轮对话始终命中
- **layer 0**：过渡区，内容相对稳定但不属于最前排
- **layer 1**：紧邻聊天历史，易受历史增长和变量变化影响

**聊天历史——最根本的缓存断裂边界**

聊天历史本身每轮必然变化（新消息追加），是所有请求中最底层的缓存断裂点。position=4 (atDepth) 的条目被注入聊天历史的指定深度，不同 depth 的稳定性截然不同：

- **depth ≥ 10**：插入历史深处，大概率非常大，编写世界书时一般用于放置到对话的最前面，这里我们选择10作为分界点 → 归入 **layer -1**（缓存前缀）
- **depth < 10**：插入靠近对话末尾，随着每轮新消息追加而频繁位移 → 归入 **layer 1**（变化区）
- **其他 position**（0/1/2/3/5/6/7）：插入聊天历史之外的固定段落（角色定义前后、AN、示例对话等），本身不受历史增长影响 → 归入 **layer 0**

**排序键**（全部升序，共6键）：

| 键位 | 排序键 | 方向 | 含义 |
|---|---|---|---|
| 第1键 | `category` | 升序 (0→1) | 纯静态条目=0，动态/边界副本/补充包裹=1 |
| 第2键 | `sub_order` | 升序 (0→2) | 同 `category` 组内进一步排序：补充包裹-开=0（最外层），普通条目/边界副本=1，补充包裹-闭=2（最外层） |
| 第3键 | `stability` | 升序 (0→2) | `pos==4 && depth>=10`→0 (核心), `pos!=4`→1 (过渡), `pos==4 && depth<10`→2 (变化) |
| 第4键 | `position` | 升序 (0→7) | 同 stability 内按插入位置枚举值自然排列 |
| 第5键 | `depth_key` | 升序 (负值→0) | `pos==4` 时用 `-depth`（depth 越大越靠前），`pos!=4` 时用 0 |
| 第6键 | `_original_order` | 升序 | 其余条件相同时维持原始相对顺序 |

**操作步骤**

1. 按上述6键公式对所有条目的当前激活列表做稳定排序
2. 根据排序结果的位次重新分配 order 值（0, 1, 2, ...）
3. layer -1/stability=0 条目自然位于缓存前缀最前端，layer 1/stability=2 条目挤到聊天历史边缘的变化区

> **order 与 depth 的关系**：详见第 8 节源码验证。简单概括——order 越大越靠后（同 position 组内排序），depth 越小越靠后（atDepth 条目在聊天历史中的嵌入位置）。两者控制不同维度的位置。

### 8.1 排序函数

### 8.2 排序 → 插入 → 最终排列

```javascript
// world-info.js L5082-5095
// 注释原文: Appends from insertion order 999 to 1. Use unshift for this purpose
[...allActivatedEntries.values()].sort(sortFn).forEach((entry) => {
    const content = getRegexedString(...);
    switch (entry.position) {
        case world_info_position.before:
            WIBeforeEntries.unshift(content);  // ★ unshift: 插入数组头部
            break;
        case world_info_position.after:
            WIAfterEntries.unshift(content);
            break;
        // ...
    }
});
```

**推演**（position=before 组）：

```
处理顺序 (sortFn降序):  order=250 → order=200 → order=100

step1: order=250 → unshift → [250]
step2: order=200 → unshift → [200, 250]
step3: order=100 → unshift → [100, 200, 250]

最终 join: "100内容\n200内容\n250内容"
          ↑ 靠前 (离聊天历史远)       ↑ 靠后 (离聊天历史近)
```

### 8.3 结论

| 字段 | 值的变化方向 | 在提示词中的位置变化 | 对输出影响力 |
|---|---|---|---|
| `order` | **越大** | **越靠后** (靠近聊天历史/生成位置) | **越大** |
| `depth` (atDepth) | **越小** (depth=0=最后消息之后) | **越靠后** (靠近生成位置) | **越大** |

即：**order越大越靠后，depth越小越靠后**。两个字段的数值方向「相反」（指影响力方向）。

### 8.4 最终提示词拼接顺序

```

  ├─ position=0 (before) 条目
  │    └─ order 低的在前, order 高的在后
  ├─ 角色描述/性格/场景定义
  ├─ position=1 (after) 条目
  │    └─ order 低的在前, order 高的在后

  │              ├─ ...
```

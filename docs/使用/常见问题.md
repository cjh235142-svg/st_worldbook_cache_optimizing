# 常见问题

## Q1：原始文件会被修改吗？数据安全吗？

**不会。** 管线执行时：
1. 脚本1自动备份原始文件为 `{原名}.backup_{时间戳}.json`
2. 所有后续脚本均写入**新文件**（`_analysis.json` / `_split.json` / `_reordered.json` / `_merged.json`）
3. 原始文件始终保持不变

## Q2：为什么静态条目的 key 被清空了？

因为静态条目被改为**蓝灯** (`constant=true`)。蓝灯条目始终激活，不需要关键词匹配，key 自然失去意义。清空 key 是减少酒馆扫描开销的优化。

## Q3：为什么动态条目全部改成了 position=4 depth=0？

这是有意的缓存优化设计：

- 将**所有动态内容**统一注入到聊天历史的**最后位置**（depth=0 = 最后一条消息之后）
- 动态内容每轮都会变化 → 必须放在缓存断裂边界**之后**
- 使用 `role=1`（User 角色）是因为 DeepSeek 更关注 User 输入
- outlet 条目（position=7）例外，保持原样不变

## Q4：补充包裹（`<补充内容>`）是什么？

补充包裹是一对始终激活的条目（`key=["/.*/"]`），用于在提示词中包裹所有动态内容：

```
<补充内容>              ← 包裹开
<暗部>动态内容</暗部>
<角色A>动态关系</角色A>
</补充内容>             ← 包裹闭
```

作用：将易变内容从缓存前缀中隔离，确保前缀稳定不受动态内容影响。

## Q5：如何在酒馆（SillyTavern）中导入优化后的世界书？

1. 打开 SillyTavern → World Info → 点击导入图标
2. 选择 `_merged.json`
3. 在角色卡设置中，将角色绑定的世界书替换为优化后的
4. 确认 API 后端配置为 DeepSeek

## Q6：管线跑第二次会怎样？幂等吗？

**幂等。** 跑两次输出一致。副本/包裹有防重复机制：

- 边界副本：检测到已有 `key=["/.*/"]` 的条目后跳过
- 补充包裹：检测到已有 `comment` 含 `[supplement-` 的条目后跳过
- 拆分标记：`[split-static]` 不会再追加

## Q7：验证工具报 XML 错误，正常吗？

**拆分和合并后的世界书 XML 标签已自动平衡**（管线在拆分时递归处理所有嵌套标签的闭标签切分和重包裹），仅以下情况会报错：

| 错误类型 | 示例 | 原因 |
|----------|------|------|
| 原始数据未闭合 | `<餐饮>`、`<连锁店>` | 源世界书本身缺少闭标签，非管线引入 |
| 宏 / HTML 误报 | `<user>`、`<br>` | `{{user}}` 宏或 HTML 标签，非 XML，可安全忽略 |

所有其他之前常见的大量跨条目 XML 断裂（如 `</背景事件>`、`</操齿凉子>` 等）均已在解析阶段自动修复。

## Q8：我想逐步手动控制，而不是一键管线？

可以逐脚本运行：

```bash
# 步骤1：分析（可手动编辑 _analysis.json 跳过不想拆的条目）
python -m scripts.analyze_entries --input 世界书.json

# 步骤2：拆分
python -m scripts.split_entries --world-book 世界书.json --analysis 世界书_analysis.json

# 步骤3：重排序
python -m scripts.reorder_entries --input 世界书_split.json

# 步骤4：合并
python -m scripts.merge_entries --input 世界书_reordered.json
```

在步骤1和2之间，编辑 `_analysis.json` 将特定条目的 `suggested_split: true` 改为 `false` 即可跳过拆分。

## Q9：outlet（position=7）条目为什么不被处理？

outlet 条目依赖 `{{outlet::Name}}` 宏显式引用，不参与自动注入。拆分会破坏 outlet 语义。整个管线对 outlet 条目保持原样：不拆分、不重排 position、不合并。

## Q10：合并后的静态条目如何追溯来源？

合并条目的 comment 格式为 `"合并:{min_order}-{max_order}"`，记录了合并前的 order 范围。可对照 `_reordered.json` 找到合并前的条目列表。

## Q11：优化后静态内容长度是否等价于原文？

语义等价但文本可能略有细微差异：
- 合并时用 `\n\n` 连接多条目 → 引入了额外空行
- 酒馆最终渲染时效果与原文一致

## Q12：为什么 `{{time}}` / `{{date}}` / `{{idleDuration}}` 被判定为动态？

这些宏每次请求展开值都不同，会破坏缓存前缀。因此被标记为动态标记，含它们的条目会被移到变化区。

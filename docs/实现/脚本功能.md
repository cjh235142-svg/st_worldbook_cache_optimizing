# 脚本功能规格

## 文件结构

```
scripts/
├── world_book_utils.py      # 公用模块
├── analyze_entries.py       # 脚本1：条目分析器
├── split_entries.py         # 脚本2：条目拆分器
├── reorder_entries.py       # 脚本3：重排序与重映射
├── merge_entries.py         # 脚本4：静态条目合并器
└── run_pipeline.py           # 主控管线（可选串联执行）
```

---

## 执行管线

```
原始世界书.json
    │
    ▼
[脚本1: analyze]  ← 自动备份原始文件
    │
    ▼ {原名}_analysis.json（用户可修改 suggested_split 字段）
    │
    ▼
[脚本2: split]  ← 默认全自动，用户可逐条控制
    │
    ▼ {原名}_split.json
    │
    ▼
[脚本3: reorder]
    │
    ▼ {原名}_reordered.json
    │
    ▼
[脚本4: merge]
    │
    ▼ {原名}_merged.json（最终输出）
```

### 产物链

```
{世界书}.json.backup_{YYYYMMDD_HHMMSS}.json    ← 原始备份
{世界书}_analysis.json                          ← 脚本1输出
{世界书}_split.json                             ← 脚本2输出
{世界书}_reordered.json                         ← 脚本3输出
{世界书}_merged.json                            ← 脚本4最终输出
```

### 主控使用

```bash
# 全流程一键执行
python run_pipeline.py --input 学园都市0313.json
```

## 公用模块 `scripts/world_book_utils.py`

| 函数/类 | 职责 |
|---|---|---|
| `load_world_book(path)` | 读 JSON，返回 entries 字典 |
| `save_world_book(entries, path)` | 写 JSON，附带备份到相同目录 |
| `backup_file(path)` | 复制文件到 `{原名}.backup_{YYYYMMDD_HHMMSS}.json` |
| `reassign_uids(entries)` | 从 0 起连续重分配所有 uid |
| `detect_markers(content)` | 返回匹配到的动态标记类型列表 |
| `classify_entry(content)` | 综合三态判定，返回 `(is_static, is_mixed)`，考虑 EJS 复合块覆盖率 |
| `sort_entries(entries)` | 按 `(is_static, stability, position, depth_key, original_order)` 升序排序，详见脚本3 |
| `find_xml_tags(content)` | 识别成对 XML 包裹标签（如 `<御坂美琴>...</御坂美琴>`） |
| `split_by_headings(content)` | 按 `#`/`##` 标题边界切分内容段落 |
| `parse_content_blocks(content)` | 将 content 按 `<%...%>` 边界切分为 text/ejs 交替块序列 |
| `find_ejs_compound_ranges(content)` | 扫描 EJS 块，追踪 brace 深度，返回 EJS 复合块的行范围列表 |
| `count_net_braces(ejs_code)` | 统计 EJS 代码块中 `{` 和 `}` 的差值（排除字符串内括号） |
| `_is_line_in_ranges(line, ranges)` | 判断行号是否落在任一 `[start, end]` 区间内 |
| `_content_fully_covered(lines, ranges)` | 检查 content 是否被 EJS 复合块完全覆盖 |
| `_strip_ejs_and_macros(content)` | 移除所有 EJS 块 `<%...%>` 和宏 `{{...}}`，返回纯文本 |
| `_remove_string_literals(code)` | 移除 JavaScript 代码中的字符串/模板/正则字面量 |
| `get_default_entry_fields()` | 返回条目的默认值字典 |

### 动态标记检测规则

```
EJS 语法：       <%  |  <%=  |  <%-
酒馆变量宏：     {{getvar::  |  {{setvar::  |  {{incvar::
小白X宏：       {{xbgetvar_yaml_idx::
条件逻辑：       {{if  |  </%if%>  |  <% if
实验性宏简写：   {{.  |  {{$
变量引用：       variables.  |  getvar(  |  setvar(
```

判定逻辑：
- 命中任意一条 → `is_static = False`（动态）
- 未命中全部 → `is_static = True`（静态）
- 命中部分但非全覆盖 → `is_mixed = True`

## 脚本 1：条目分析器 `scripts/analyze_entries.py`

| 项 | 内容 |
|---|---|
| **输入** | `--input <世界书.json>` |
| **备份** | 执行前自动备份原文件 |
| **处理** | 遍历 entries，对每条 content 扫描动态标记，判定 `is_static` / `is_mixed` |
| **输出** | `{原名}_analysis.json` |

### 输出 JSON 结构

```json
{
  "source": "学园都市0313.json",
  "generated_at": "2026-06-01T12:00:00",
  "entries": [
    {
      "uid": 5,
      "comment": "核心01：御坂美琴",
      "is_static": false,
      "is_mixed": true,
      "detected_markers": ["<%", "getvar"],
      "suggested_split": true
    },
    {
      "uid": 8,
      "comment": "事件：背景",
      "is_static": false,
      "is_mixed": true,
      "detected_markers": ["<%", "getvar"],
      "suggested_split": true
    }
  ],
  "summary": {
    "total": 100,
    "static": 60,
    "dynamic": 18,
    "mixed": 22
  }
}
```

### 说明

- `suggested_split` 仅对 `is_mixed=true` 的条目设为 `true`
- 用户可编辑 `_analysis.json`，将不希望拆分的条目改为 `"suggested_split": false`

## 脚本 2：条目拆分器 `scripts/split_entries.py`

| 项 | 内容 |
|---|---|
| **输入** | `--world-book <世界书.json> --analysis <分析结果.json>` |
| **处理** | 对 `suggested_split=true` 的条目，沿 Markdown 标题边界 / XML 子标签边界 / EJS 条件块边界拆分 |
| **输出** | `{原名}_split.json` |

### 拆分规则

1. 识别 content 中的**最外层 XML 包裹标签对**（如 `<御坂美琴>...</御坂美琴>`）
2. 按 `#` / `##` 标题层级切分内容
3. 各切片分为以下两类：
   - **静态切片**：不含任何动态标记 → 组装为静态子条目
   - **动态切片**：包含动态标记 → 组装为动态子条目
4. 每个子条目各自包裹原有的最外层 XML 标签（合并后语义等价于原文）
5. 子条目 key/keysecondary 继承原条目
6. 子条目 comment 追加标识：

| 类型 | comment 后缀 |
|---|---|
| 静态 | `[split-static]` |
| 动态 | `[split-dynamic]` |

### 拆分后 uid 分配

拆分后全量条目从 0 起重新连续分配 uid。

### 不可拆分的情况（跳过）

- 纯 EJS 全覆盖（整个 content 都在 `<% if/else %>` 分支中）
- 同一语义单元内静态与动态内容混排，无法在更细粒度上拆分

## 脚本 3：重排序与重映射 `scripts/reorder_entries.py`

| 项 | 内容 |
|---|---|
| **输入** | `--input <拆分后世界书.json>` [`--wrapper-name <名称>`] |
| **输出** | `{原名}_reordered.json` |

### 规则表

| 条件 | 修改字段 |
|---|---|
| 静态 + `pos==4` + `depth<10` | `position=4, depth=0, role=1, constant=true, cooldown=0, probability=100, useProbability=false, sticky=0, delay=0` |
| 静态 + `pos==4` + `depth>=10` | `position=4, depth=9999, role=1, constant=true, cooldown=0, probability=100, useProbability=false, sticky=0, delay=0` |
| 静态 + `pos!=4` | `constant=true, cooldown=0, probability=100, useProbability=false, sticky=0, delay=0`（pos/depth/role 保持原值） |
| 动态（全部） | `position=4, depth=0, role=1`（constant/cooldown/probability/useProbability/sticky/delay 保持原值） |
| 边界副本 / 补充包裹 | `position=4, depth=0, role=1, constant=false, key=["/.*/"], useProbability=false, probability=100, cooldown=0, sticky=0, delay=0` |

### 排序与 order 分配

**排序键**（全部升序）：

| 键位 | 排序键 | 方向 | 含义 |
|---|---|---|---|
| 第1键 | `is_static` | 升序 (0→1) | 静态(0)全部在前，动态(1)全部在后 |
| 第2键 | `stability` | 升序 (0→2) | `pos==4 && depth>=10`→0(核心), `pos!=4`→1(过渡), `pos==4 && depth<10`→2(变化) |
| 第3键 | `position` | 升序 (0→7) | 同层内按插入位置枚举值自然排列 |
| 第4键 | `depth_key` | 升序 | `pos==4` 时用 `-depth`（depth 越大越靠前）；`pos!=4` 时用 `0`（depth 不参与排序） |
| 第5键 | `original_order` | 升序 | 其余条件相同时维持原始相对顺序 |

排序后从 0 递增分配新 order。由于排序第1键已保证静态全部在前、动态全部在后，无需额外 offset：

```
排序后: [static, ..., static, dynamic, ..., dynamic]
order:    0      S-1     S          S+D-1

min(动态.order) = S > S-1 = max(静态.order)  自然满足
```

## 脚本 4：静态条目合并器 `scripts/merge_entries.py`

| 项 | 内容 |
|---|---|
| **输入** | `--input <重排后世界书.json>` |
| **输出** | `{原名}_merged.json`（最终输出） |

### 合并规则

| 维度 | 处理 |
|---|---|
| **分组条件** | `(position, constant, depth)` 三者完全相同的**可合并静态**条目 |
| **排除项** | 动态条目（含动态标记）、边界副本（comment 含 `[boundary-copy-`）、补充包裹（comment 含 `[supplement-`）不参与合并 |
| **content** | 按 order 升序，用 `\n\n` 拼接 |
| **order** | 取组内最小 order |
| **key** | 清空为 `[]`（合并后关键词实际不起作用） |
| **keysecondary** | 清空为 `[]` |
| **comment** | 设为 `"合并:{min_order}-{max_order}"` |
| **其他字段** | 取默认值（与酒馆创建新条目的默认值一致） |
| **uid** | 合并后全量条目从 0 起重新连续分配 |

### 动态条目

完全不参与合并，保持原样。但最终 uid 重分配时仍参与（与合并后的静态条目统一排序）。

### 合并前 vs 合并后（示例）

```
合并前：
  条目 uid=3  content="<设定>A的设定"    pos=4 depth=0 constant=true order=0
  条目 uid=7  content="<设定>B的设定"    pos=4 depth=0 constant=true order=1
  条目 uid=12 content="<设定>C的设定"    pos=4 depth=0 constant=true order=2
  条目 uid=5  content="<% getvar() %>"  pos=4 depth=0 constant=false order=5

合并后：
  条目 uid=0  content="<设定>A的设定\n\n<设定>B的设定\n\n<设定>C的设定"
              pos=4 depth=0 constant=true order=0
              comment="合并:0-2"  key=[]  keysecondary=[]
  条目 uid=1  content="<% getvar() %>"  pos=4 depth=0 constant=false order=5
```

# 集成测试方案

## 测试入口

各脚本暴露 Python API，通过 pytest 直接串联调用（非 subprocess）：

```bash
# 全部测试（单元 + 集成）
pytest scripts/tests/ -v

# 仅集成测试
pytest scripts/tests/test_integration.py -v
```

---

## 文件结构

```
scripts/tests/
├── conftest.py                          # 追加集成测试 fixture
├── fixtures/
│   └── integration/
│       ├── full_pipeline_input.json     # 完整管线输入：13 条覆盖所有场景
│       ├── full_pipeline_expected.json  # Golden File：人工构造的预期最终产物
│       ├── all_static.json              # 全静态世界书
│       ├── all_dynamic.json             # 全动态世界书
│       ├── single_entry.json            # 单条目
│       └── empty_entries.json           # 空 entries
└── test_integration.py
```

## 各脚本需暴露的 API

```python
# analyze_entries.py
def run(input_path: str, output_path: str | None = None) -> str:
    """返回 analysis JSON 文件路径"""

# split_entries.py
def run(world_book_path: str, analysis_path: str,
        output_path: str | None = None) -> str:
    """返回拆分后世界书路径"""

# reorder_entries.py
def run(input_path: str, output_path: str | None = None) -> str:
    """返回重排后世界书路径"""

# merge_entries.py
def run(input_path: str, output_path: str | None = None) -> str:
    """返回最终世界书路径"""
```

所有函数在 `output_path=None` 时自动生成默认输出路径。

## `full_pipeline_input.json` 设计

覆盖 13 条，确保管线每步都有实际数据可处理：

| 条目 | 条件 | 拆分预期 | 重排预期 | 合并预期 |
|---|---|---|---|---|
| E1 | 静态 pos=4 depth=3 | — | → 用户 depth=0 蓝灯 | 与 E2 合并 |
| E2 | 静态 pos=4 depth=5 | — | → 用户 depth=0 蓝灯 | 与 E1 合并 |
| E3 | 静态 pos=4 depth=10 | — | → 用户 depth=9999 蓝灯 | 与 E4 合并 |
| E4 | 静态 pos=4 depth=15 | — | → 用户 depth=9999 蓝灯 | 与 E3 合并 |
| E5 | 静态 pos=0 | — | → 蓝灯，pos 不变 | 单条不合并 |
| E6 | 静态 pos=1 | — | → 蓝灯，pos 不变 | 单条不合并 |
| E7 | 静态 pos=6 | — | → 蓝灯，pos 不变 | 单条不合并 |
| E8 | 动态 pos=0 | 不拆分 | → pos=4 depth=0 role=1 保持绿灯 | 不合并 |
| E9 | 动态 pos=4 depth=5 | 不拆分 | → pos=4 depth=0 role=1 保持绿灯 | 不合并 |
| E10 | 动态 pos=1 | 不拆分 | → pos=4 depth=0 role=1 | 不合并 |
| E11 | 混合可拆分（XML+H2） | 拆为2条 | 静态→蓝灯，动态→pos4 | 静态参与合并 |
| E12 | 混合可拆分（XML+##） | 拆为2条 | 静态→蓝灯，动态→pos4 | 静态参与合并 |
| E13 | 混合不可拆分（全EJS覆盖） | 不拆分 | → pos=4 depth=0 role=1 | 不合并 |

## `full_pipeline_expected.json` Golden File 人工构造要点

已知输入 13 条，按规则推演最终条目数：

| 阶段 | 条目数 | 说明 |
|---|---|---|
| 输入 | 13 | |
| 脚本1 分析 | 13 | 产出 analysis |
| 脚本2 拆分 | 15 | E11/E12 各拆为2条（+2），E13 拆不了 |
| 脚本3 重排 | 17 | 新增补充包裹2条，全排序 + 字段改写 |
| 脚本4 合并 | 13 | E1+E2 合并(-1)，E3+E4 合并(-1)，其余同组静态合并(-2)；后重新分配 order |

最终条目结构逐字段手工构造，跑通后 snap 确认。

## 测试用例清单 (29 用例)

### 1. 全管线端到端 (5 用例)

| # | 用例 | 输入 fixture | 验证点 |
|---|---|---|---|
| I1 | 完整四步管线执行成功 | `full_pipeline_input.json` | 四步全部无异常，最终产物为合法 JSON |
| I2 | 产物链文件完整性 | 同上 | 每步输出文件均存在 + 备份文件存在 |
| I3 | 全静态世界书 | `all_static.json` | 不报错，正常输出 |
| I4 | 全动态世界书 | `all_dynamic.json` | 不报错，动态 offset 正确（无静态时 offset=1） |
| I5 | 单条目世界书 | `single_entry.json` | 不报错不崩溃，输出 1 条 |

### 2. Golden File 精确比对 (1 用例)

| # | 用例 | 输入 | 验证点 |
|---|---|---|---|
| I6 | 输出与 Golden File 逐字段匹配 | `full_pipeline_input.json` | `_merged.json` 每个条目每个字段 ≡ `full_pipeline_expected.json` |

### 3. 语义等价性 (3 用例)

| # | 用例 | 输入 | 验证点 |
|---|---|---|---|
| I7 | 原始 content 无丢失 | `full_pipeline_input.json` | 每个原始条目的 content 都能在最终产物的某条 content 中找到完整子串 |
| I8 | 动态 content 完整性 | 同上 | 所有动态条目的原 content 在最终产物中逐字保留（未被合并破坏） |
| I9 | 拆分语义可逆 | 混合条目 E11/E12 | 拆分后的静态 + 动态 content 拼接 ≈ 原 content（忽略标签重复） |

### 4. Order 约束验证 (3 用例)

| # | 用例 | 输入 | 验证点 |
|---|---|---|---|
| I10 | 静态 order < 动态 order | `full_pipeline_input.json` | `max(static_order) < min(dynamic_order)` |
| I11 | 同组原始相对顺序保持 | 同上 | 合并前同 `(pos, constant, depth)` 组内 order 升序与 content 拼接顺序一致 |
| I12 | order 连续无空洞 | 同上 | 最终产物所有 order 值从 0 起集合连续 |

### 5. 字段修改验证 (4 用例)

| # | 用例 | 输入 | 验证点 |
|---|---|---|---|
| I13 | 静态全部蓝灯 | `full_pipeline_input.json` | 所有静态条目 `constant=true` |
| I14 | 静态概率检查关闭 | 同上 | 所有静态条目 `useProbability=false probability=100` |
| I15 | 静态冷却清零 | 同上 | 所有静态条目 `cooldown=0 sticky=0 delay=0` |
| I16 | 动态全部 pos=4 depth=0 role=1 | 同上 | 所有动态条目 `position=4 depth=0 role=1` |

### 6. 合并验证 (3 用例)

| # | 用例 | 输入 | 验证点 |
|---|---|---|---|
| I17 | 同组静态去重 | `full_pipeline_input.json` | 最终产物中不存在两条 `(pos, constant, depth)` 完全相同的静态条目 |
| I18 | 动态条目未合并 | 同上 | 动态条目数量 = 脚本 3 输出中动态条目数量 |
| I19 | 静态 key 已清空 | 同上 | 所有静态条目 `key=[] keysecondary=[]` |

### 7. 幂等性 (2 用例)

| # | 用例 | 输入 | 验证点 |
|---|---|---|---|
| I20 | 二次管线语义一致 | 第一次最终产物 `_merged.json` 作为第二次输入 | 两次 `_merged.json` 逐字段完全一致 |
| I21 | 二次无重复拆分标记 | 同上 | 最终产物中无条目 comment 出现重复 `[拆分-...]` 或嵌套 `合并:...` |

### 8. 容错 (2 用例)

| # | 用例 | 输入 | 验证点 |
|---|---|---|---|
| I22 | 空 entries | `empty_entries.json` | 四步均不报错，最终输出 `entries: {}` |
| I23 | 无 analysis 文件时拆分行为 | 不提供 analysis 文件 | 明确报错（行为一致即可） |

### 边界副本与补充包裹 (5 用例)

| # | 用例 | 输入 | 验证点 |
|---|---|---|---|
| I24 | 管线—边界副本创建且不合并 | 世界书含 `<暗部>`+ 动态条目 + `</暗部>` | `_merged.json` 中存在 `key=["/.*/"]` 的边界副本，且副本未被合并到其他静态条目中 |
| I25 | 管线—边界无动态不创建 | 世界书含 `<暗部>`+ 全静态 + `</暗部>` | `_merged.json` 中不存在对应的边界副本条目 |
| I26 | 管线—补充包裹存在且不合并 | 世界书含任意动态条目 | `_merged.json` 中存在 `comment="[supplement-start]"` / `"[supplement-end]"` 条目，且包裹未被合并到其他条目 |
| I27 | 管线—全静态无补充包裹 | `all_static.json` | `_merged.json` 中不存在任何 `[supplement-*]` 条目 |
| I28 | 管线—包裹内容语义完整 | 含动态条目的世界书 | 开包裹在动态区最外层（order 最小），闭包裹在动态区最外层（order 最大），全部动态条目和边界副本在两者之间 |

## conftest.py 追加内容

```python
# 追加到 scripts/tests/conftest.py

INTEGRATION_FIXTURES_DIR = FIXTURES_DIR / "integration"


@pytest.fixture
def integration_fixtures_dir() -> Path:
    return INTEGRATION_FIXTURES_DIR


@pytest.fixture
def full_pipeline_input(integration_fixtures_dir) -> dict:
    return load_fixture(str(integration_fixtures_dir / "full_pipeline_input.json"))


@pytest.fixture
def full_pipeline_expected(integration_fixtures_dir) -> dict:
    return load_fixture(str(integration_fixtures_dir / "full_pipeline_expected.json"))


@pytest.fixture
def pipeline_tmpdir(tmp_path) -> Path:
    """管线各步输出到临时目录，避免污染源目录"""
    d = tmp_path / "pipeline_test"
    d.mkdir(exist_ok=True)
    return d
```

## CI 执行方式

```bash
# 全部测试（单元 + 集成）
pytest scripts/tests/ -v

# 仅单元测试
pytest scripts/tests/ --ignore=scripts/tests/test_integration.py -v

# 仅集成测试
pytest scripts/tests/test_integration.py -v
```

`pyproject.toml` 中配置：

```ini
[tool.pytest.ini_options]
testpaths = ["scripts/tests"]
```
